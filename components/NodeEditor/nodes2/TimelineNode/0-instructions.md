# Жесткие правила взаимодействия для SpriteForge

1. **Принцип минимального вмешательства:** Изменяй только те строки кода, которые напрямую связаны с выполнением текущей задачи. Запрещено переписывать соседние функции, менять форматирование или исправлять стиль кода, если об этом явно не просили.
2. **Запрет на документацию изменений:** Никогда не добавляй в код комментарии, поясняющие твои действия (например, `// Changed this`, `// STRICTLY RED`). Не изменяй и не удаляй существующие комментарии автора.
3. **Нулевая визуальная инициатива:** Если дано указание изменить цвет конкретного элемента (например, «UI тестов — синий»), это касается только этого элемента. Запрещено «подгонять» остальные элементы под этот стиль для «целостности дизайна».
4. **Протокол предложений (Initiative Proposal):** Если ты видишь баг, возможность оптимизации или логическую нестыковку, которую пользователь не заметил:
    * ЗАПРЕЩЕНО вносить исправления в код сразу.
    * ОБЯЗАН написать краткий список предложений в текстовом ответе.
    * ЖДАТЬ явного слова «ОДОБРЯЮ» для каждого пункта перед редактированием файлов.
5. **Точность именования и типов:** Не меняй названия переменных, ID или структуру объектов в `types.ts`, если это не является целью задачи.
6. **Отчетность:** В текстовом ответе перед XML-блоком пиши только сухую спецификацию: что именно изменено и в каких файлах. Без извинений, оправданий и лишних рассуждений.
