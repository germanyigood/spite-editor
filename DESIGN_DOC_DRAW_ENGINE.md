# 🎨 Дизайн-документ: Продвинутый движок рисования (Photoshop-grade Draw Engine)

## 1. Введение и цели
Текущая реализация режима `draw` (в частности, в `useDrawingEngine.ts` и `PaintNode`) обладает рядом фундаментальных ограничений: она использует базовый `CanvasRenderingContext2D` и рисует независимые круги (`ctx.arc`) на события `pointermove`. Это приводит к "рваным" линиям при быстром движении мыши (отсутствует сплайновая интерполяция), игнорирует степень нажатия стилуса (Pointer Events API) и не поддерживает сложные инструменты (фигуры, пути, слои). 

**Цель:** Спроектировать архитектуру движка рисования профессионального уровня (Photoshop, Procreate), который позволит пользователям производить сложные манипуляции с графикой прямо внутри AI Studio без необходимости экспорта в сторонние редакторы.

---

## 2. Архитектура движка (Core Render Pipeline)

Для достижения высокой производительности и поддержки сложных эффектов, рендеринг должен быть разделен на несколько логических слоев:

1. **Document State (DOM-независимый):** 
   * Состояние Paint-ноды больше не является одной плоской Bitmap картинкой (`paintData`).
   * Состояние должно быть представлено объектом `LayeredDocument`, содержащим упорядоченный массив `Layer` (Растровые и Векторные слои).
2. **View Canvas (UI):** Главный `<canvas>`, который отображается пользователю. Занимается исключительно отрисовкой `LayeredDocument` с учетом текущего масштаба, панорамирования (Pan/Zoom) и наложения UI-элементов (Marching Ants для выделения, Transform контролы).
3. **Working / Overlay Canvas:** Отдельный слой (offscreen или `<canvas>` поверх View Canvas), предназначенный для отрисовки "активного" действия. Например, пока пользователь ведет кистью, штрих рисуется здесь. По завершении действия (`pointerup`) штрих запекается (bake) в активный слой (через compositing).

### 2.1 Undo/Redo Engine (Командный паттерн)
Для Photoshop-like опыта необходима мощная история:
* Вместо сохранения полных Blob-снимков на каждый чих (что убьет память), необходимо использовать паттерн Command.
* Действия (`DrawStrokeCommand`, `FillCommand`, `MoveLayerCommand`) применяются и откатываются атомарно.
* Растровые изменения (штрихи) сериализуются как дифы (разница пикселей) или сохраняется кэш слоя до/после изменения.

## 3. Движок кистей (Advanced Brush Engine)

Текущий алгоритм — "рисование кругов по дискретным точкам" — морально устарел.

### 3.1 Сглаживание и интерполяция (Spline Smoothing)
1. **Сбор точек (Input):** Собираем исходные точки `{x, y, pressure}` из `PointerEvent`.
2. **Сглаживание (Smoothing):** Используем алгоритм натяжения сплайна (**Catmull-Rom Spine** или **Bezier Curve**), чтобы аппроксимировать траекторию между сырыми точками от мыши.
3. **Интерполяция:** Идем по сплайну с заданным шагом (`spacing` кисти) и генерируем точки для штришков (dabs).

### 3.2 Характеристики кисти (Brush Dynamics)
Каждая кисть (включая ластик) должна иметь набор параметров:
* **Size (Размер) & Hardness (Жесткость):** Жесткость определяет размытие краев круга, используя радиальный градиент.
* **Opacity (Непрозрачность) & Flow (Нажим):** Flow работает накопительно, если штрих проходит по одному и тому же месту, в то время как Opacity ограничивает максимальную плотность слоя в рамках одного штриха.
* **Spacing:** Расстояние между "отпечатками" (dabs) кисти в процентах.
* **Shape / Texture:** Возможность использовать альфа-маску вместо сплошного круга (текстурные кисти).
* **Pressure Sensitivity:** Привязка `e.pressure` (от планшетов типа Wacom/iPad) к Size или Opacity. Толщина линии должна плавно меняться (использование формул вроде `size = baseSize * (minPressure + (1 - minPressure) * e.pressure)`).

## 4. Расширенный набор инструментов (Toolbox)

### 4.1 Растровые инструменты
* **Brush, Pencil, Eraser:** На базе нового Brush Engine. Режим Pencil использует алгоритм Брезенхема (без антиалиасинга) для пиксель-арта.
* **Paint Bucket (Заливка):** Использование алгоритма Flood Fill (на CPU-массиве `Uint8ClampedArray` из `ImageData`) с поддержкой Tolerance (допуска совпадения цвета) и Contiguous/Global режимами.
* **Smudge / Blur / Dodge / Burn:** Инструменты ретуши. Требуют чтения пикселей под кистью, применения матрицы свертки (convolution matrix) или цветовой трансформации и возврата их на холст.

### 4.2 Векторные инструменты (Шейпы и Пути)
Paint-нода должна стать гибридной (Vector + Raster).
* **Pen Tool (Перо):** Рисование путей (SVG Path) через кривые Безье, манипуляция контрольными точками (Anchor Points, Handles).
* **Shapes (Фигуры):** Инструменты Rectangle, Ellipse, Polygon.
Реализация: Фигуры рисуются на математическом (векторном) слое и остаются редактируемыми, пока пользователь не "растеризует" слой.

### 4.3 Выделение (Selections)
* **Marquee (Прямоугольное, Овальное) & Lasso (Произвольное, Полигональное).**
* **Magic Wand (Волшебная палочка):** Алгоритм Flood Fill, возвращающий маску выделения.
* **Selection Mask:** Выделение - это самостоятельный 8-битный grayscale канал (OffscreenCanvas). Любой инструмент рисования перед применением к слою должен быть clip'нут (умножен) по маске выделения (`ctx.globalCompositeOperation = 'in'`).
* **Marching Ants:** Анимация пунктирной линии по границам маски выделения.

### 4.4 Трансформации (Free Transform)
* Позволяет брать выделенную область (или весь слой) и применять к ней Translation, Rotation, Scaling и Skew.
* UI: 8 точек (handles) по краям Bounding Box'а. Углы крутят или масштабируют элементы с соблюдением пропорций (`Shift`).
* Матрица трансформации применяется к пикселям (либо векторному пути) перед окончательным запеканием.

## 5. Система слоев и композитинг (Layers & Compositing)

Внутри одного холста (PaintNode) должна быть локальная мини-система слоев:
* **Blend Modes:** Поддержка `Multiply`, `Screen`, `Overlay`, `Color Dodge` и т.д. В Canvas API это соотносится с `ctx.globalCompositeOperation`.
* **Opacity:** Глобальная прозрачность слоя.
* **Visibility / Lock:** Видимость слоя и защита от редактирования.
* **Clipping Masks:** Возможность обрезать один слой по альфа-каналу слоя ниже.

## 6. UI / UX Изменения

Для интеграции в текущий Layout Editor и Sprite Editor потребуется:
1. **Tool Toolbar (Левая или Верхняя панель):**
   * Выбор активного инструмента (Move, Marquee, Lasso, Wand, Crop, Brush, Pencil, Stamp, Eraser, Fill, Text, Shapes).
2. **Contextual Settings (Верхний бар):**
   * При выбранной кисти: слайдеры Size, Hardness, Opacity. Иконка включения Pressure.

3. **Layer Panel (Правая панель):**
   * Миниатюры слоев, кнопки Add, Delete, Duplicate. Dropdown режимов наложения. Глазик (видимость).

## 7. План внедрения (Roadmap)

Внедрение такого масштаба следует делать по фазам, чтобы не сломать текущий пайплайн `NodeGraph`.

### Фаза 1: Улучшение Brush Engine (Quick Win)
* Заменить текущую логику `useDrawingEngine.ts` на использование буферного слоя (Working Canvas).
* Добавить интерполяцию сплайнами (Catmull-Rom), чтобы штрих всегда был плавным, вне зависимости от частоты кадров мыши.
* Добавить поддержку давления со стилуса (`e.pressure`).

### Фаза 2: Базовые инструменты
* Внедрить Paint Bucket Tool (Алгоритм Flood Fill с Tolerance).
* Добавить простые геометрические фигуры (Rect, Ellipse), которые растеризуются при отпускании мыши.
* Добавить ластик нормального качества с поддержкой мягких краев (Hardness).

### Фаза 3: Система выделений
* Реализовать Selection Mask.
* Инструменты Marquee и Magic Wand.
* Ограничение области рисования (Clipping) границами активного выделения.

### Фаза 4: Слои внутри Paint Node и Трансформации
* Обновить семантику `paintData` в схеме нод - переход от одного бинарного blob_url к JSON-структуре (массив слоев + blob_url для каждого слоя).
* Инструмент Free Transform для перемещения нарисованных слоев или выделенных сегментов.
* Внедрение UI для управления слоями локально внутри Inspector'а ноды.

## Заключение
Данный дизайн смещает акцент с примитивного "рисования поверх узла" на полноценное создание контента (Content Creation Tool). Перевод пайплайна на буферную адресацию, векторно-математическое сглаживание штрихов и слоистую композицию сделает приложение универсальным инструментом манипуляций, не уступающим Photoshop в базовых задачах редактирования текстур и спрайтов.

## 8. Пользовательские истории (User Stories)

### 8.1 Рисование и Кисти (Brush Engine)
* **US-BRUSH-01:** Как художник, я хочу быстро вести кистью по холсту и получать абсолютно гладкую линию без углов и разорванных кругов, чтобы мои линейные рисунки (Lineart) выглядели профессионально.
* **US-BRUSH-02:** Как пользователь графического планшета (Wacom/iPad), я хочу, чтобы ширина и прозрачность кисти зависели от силы нажатия пера (`e.pressure`), чтобы я мог создавать живые штрихи с естественным сужением на концах.
* **US-BRUSH-03:** Как пиксель-артист, я хочу использовать инструмент Pencil с жесткостью 100% и размером в 1px без anti-aliasing (сглаживания), чтобы каждый пиксель ставился четко и не "мылил" соседние пиксели.
* **US-BRUSH-04:** Как ретушер, я хочу использовать кисть с мягкими краями (Hardness 0-50%), чтобы аккуратно закрашивать или стирать границы объектов без резких переходов.

### 8.2 Инструменты и Выделение (Tools & Selections)
* **US-TOOL-01:** Как дизайнер, я хочу выделить определенный элемент с помощью Magic Wand (Волшебной палочки), настроив Tolerance, чтобы быстро менять его цвет, не задевая фон.
* **US-TOOL-02:** Как пользователь, я хочу нарисовать фигуру (Прямоугольник/Эллипс) и закрасить её область с помощью Bucket Fill в один клик, чтобы не заштриховывать её вручную.
* **US-TOOL-03:** Как ретушер, я хочу выделить зону с помощью инструмента Lasso, чтобы при последующем рисовании кистью краска ложилась *только* внутри выделенной области (Selection Mask/Clipping).

### 8.3 Слои и Композитинг (Layers & Compositing)
* **US-LAYER-01:** Как цифровой художник, я хочу создавать новый пустой слой для наброска, а затем рисовать поверх него чистовик на другом слое, чтобы в конце просто скрыть слой с наброском.
* **US-LAYER-02:** Как колорист, я хочу поменять режим наложения (Blend Mode) слоя на `Multiply` и уменьшить его Opacity до 50%, чтобы покрасить тени поверх базового цвета, не закрашивая контур.
* **US-LAYER-03:** Как пользователь, я хочу менять порядок слоев (Drag & Drop), чтобы переносить элементы на передний или задний план композиции.

### 8.4 Трансформация и История (Transform & History)
* **US-TRANSFORM-01:** Как креатор, я хочу выделить часть холста и использовать Free Transform (масштабирование и поворот), чтобы скорректировать пропорции отдельного элемента без перерисовки.
* **US-TRANSFORM-02:** Как любой пользователь, я хочу иметь возможность нажать `Ctrl+Z` (Undo) более 50 раз, чтобы отменить неудачную цепочку штрихов, и `Ctrl+Shift+Z` (Redo), чтобы вернуть их, если я передумал.

## 9. Стратегия тестирования и крайние случаи (Testing & Corner Cases)

Согласно правилу `Один файл — один тест` и избеганию "ленивого" тестирования, описываемые тест-кейсы охватывают глубокие проверки состояний и корнер-кейсы. В E2E тестах (например, в Playwright или кастомном `Simulator.ts`) должны использоваться исключительно `data-testid` (никаких CSS-селекторов).

### 9.1 Тестирование Brush Engine (`brushPhysics.spec.ts`)
*   **Case 1: Скоростная интерполяция.** 
    *   *Действие:* Имитация движения мыши (PointerEvent) между `(0,0)` и `(1000, 1000)` всего за 2 кадра (события).
    *   *Ожидание:* Движок должен сгенерировать сплайн между точками и отрисовать непрерывную линию пикселей (dab spacing), а не две изолированные точки.
*   **Case 2: Сверхчувствительность пера (Pressure 0.01 vs 1.0).**
    *   *Действие:* Штрих с плавно меняющимся значением `pressure` от 0.01 до 1.0.
    *   *Ожидание:* Толщина линии в начале штриха равна `minSize`, в конце штриха точно равна `maxSize`.
*   **Case 3: Отрисовка вне границ холста (Out of Bounds).**
    *   *Действие:* Пользователь начинает штрих внутри (50,50), уводит мышь за пределы `<canvas>` (-100, -100) и возвращает обратно.
    *   *Ожидание:* События вне холста (захваченные через `setPointerCapture`) корректно интерполируются. Линия не обрывается на границе, а уходит за нее и возвращается без визуальных глитчей.
*   **Case 4: Нулевая дистанция (Dot Click).**
    *   *Действие:* События `pointerdown` и сразу `pointerup` в одних координатах (без `pointermove`).
    *   *Ожидание:* Должен отрисоваться идеально ровный круг (dab) заданного размера и жесткости.

### 9.2 Тестирование выделений и обтравки (`selections.spec.ts`)
*   **Case 1: Рисование на границе маски (Clipping edge case).**
    *   *Действие:* Выделить Marquee прямоугольник `(10,10)` до `(50,50)`. Провести кистью размером 20px ровно по границе `X=10`.
    *   *Ожидание:* Пиксели окрашиваются **строго** с координат `X=10` до `X=50`. Пиксели на `X=9` остаются нетронутыми (жесткий clip/masking).
*   **Case 2: Негативное выделение (Subtract Mode).**
    *   *Действие:* Выделить весь холст. Включить режим "Subtract" и выделить центр. Залить (Flood Fill) весь холст.
    *   *Ожидание:* Заливка применяется как рамка, центр остается пустым.
*   **Case 3: Инверсия пустого выделения.**
    *   *Действие:* При отсутствии выделения вызвать действие "Invert Selection".
    *   *Ожидание:* Выделяется весь холст `(0,0)` до `(width, height)`.

### 9.3 Тестирование заливки (`floodFill.spec.ts`)
*   **Case 1: Утечка через 1 пиксель (1px leak).**
    *   *Действие:* Нарисовать круг, который не замкнут всего на 1 диагональный пиксель. Произвести заливку внутрь.
    *   *Ожидание:* Заливка НЕ должна вытечь за пределы (при отключенном anti-aliasing или Tolerance = 0). Если включено сглаживание, поведение должно учитывать прозрачность "дырки" в зависимости от Tolerance.
*   **Case 2: Сверхвысокий Tolerance (Tolerance = 255).**
    *   *Действие:* Кликнуть заливкой с Tolerance 255 в любую точку холста с градиентом.
    *   *Ожидание:* Должен залиться весь холст, игнорируя любые цветовые значения оригинального изображения.
*   **Case 3: Островная заливка (Contiguous = false).**
    *   *Действие:* На холсте красный кружок слева и красный кружок справа (отделены черной линией). Кликнуть заливкой в левый кружок при `contiguous = false`.
    *   *Ожидание:* Оба красных кружка закрашиваются новым цветом синхронно.

### 9.4 Тестирование слоев (`layers.spec.ts`)
*   **Case 1: Удаление единственного активного слоя.**
    *   *Действие:* Попытка нажать `data-testid="delete-layer-btn"` на последнем оставшемся слое.
    *   *Ожидание:* Либо кнопка должна быть `disabled`, либо действие должно просто очистить слой (clear), но не удалять сам объект из массива слоев, гарантируя что длина `layers[] >= 1`.
*   **Case 2: Смена Opacity и Blend Mode (Math Verification).**
    *   *Действие:* Нижний слой залит `rgb(255, 0, 0)`. Верхний слой залит `rgb(0, 0, 255)` и переведен в `Multiply`.
    *   *Ожидание:* Результирующий композит (проверенный через `getImageData`) должен быть ровно `rgb(0, 0, 0)`.
*   **Case 3: Изоляция кликов невидимого слоя.**
    *   *Действие:* Скрыть слой (Toggle Visibility -> false). Попытаться провести по нему кистью.
    *   *Ожидание:* Движок должен выбросить предупреждение (или UI должен заблокировать действие `pointerdown`), массив пикселей не должен быть модифицирован.

### 9.5 Тестирование Истории (Undo/Redo) (`history.spec.ts`)
*   **Case 1: Разрыв ветки Redo.**
    *   *Действие:* Сделать действия A, B, C. Нажать Undo дважды (остались на A). Сделать действие D.
    *   *Ожидание:* Стек Redo (содержащий B и C) должен быть безвозвратно очищен (Disposed) во избежание утечек памяти. Доступная история: A -> D.
*   **Case 2: Атомарность перемещения слоев.**
    *   *Действие:* Переместить слой 1 выше слоя 2 (Drag & Drop в панели слоев). Нажать Undo.
    *   *Ожидание:* Слой 1 должен вернуться на свою изначальную позицию (индекс в массиве), и View Canvas должен перерисоваться.
*   **Case 3: Лимит стека памяти.**
    *   *Действие:* Установить лимит истории в 5 действий. Выполнить 7 действий.
    *   *Ожидание:* Размер стека должен оставаться равным 5, первые 2 действия безвозвратно удаляются (shift).

### 9.6 Тестирование Transformer (`transform.spec.ts`)
*   **Case 1: Отрицательный Scale (Mirroring).**
    *   *Действие:* Пользователь тянет левый Anchor за правую границу Bounding Box'а.
    *   *Ожидание:* Элемент должен зеркально отразиться (Flip Horizontal) без крашей матрицы.
*   **Case 2: Саб-пиксельный поворот.**
    *   *Действие:* Повернуть пиксель-арт (Pencil tool) на 45 градусов. 
    *   *Ожидание:* Если включен алгоритм Nearest Neighbor, обрезка пикселей должна оставаться "острой" (hard edges), без появления полупрозрачных размытых границ.

### 9.7 Требования к E2E симуляциям (`drawingE2E.spec.ts`)
В рамках E2E вызовы симулятора (`Simulator.ts`) должны точно воспроизводить UI-поток:
1. Клик по `data-testid="tool-brush"`.
2. Изменение значения в `data-testid="brush-size-input"`.
3. Удержание `PointerDown` на `data-testid="draw-canvas"`.
